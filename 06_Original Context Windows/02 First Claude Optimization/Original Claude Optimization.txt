This is a well-posed computational geometry / spatial econometrics question. Let me break it down precisely.

The Core Insight
The original process is intractable because it conflates two fundamentally different things that happen to occur together:

Topology — who is adjacent to whom — which is time-invariant (cells don't move)
Attributes — what values neighbors have — which changes every year
If your code recomputes neighbor search (e.g. poly2nb, dnearneigh, knearneigh) inside a year loop or variable loop, you're solving a geometry problem 14× (or 14 × 110×) when you only need to solve it once.

The Reformulation
Step 1 — Extract the graph once
r
library(spdep)
library(sf)

# Run ONCE — topology never changes
nb <- poly2nb(prep_data, queen = TRUE)        # adjacency list
W  <- nb2listw(nb, style = "W")              # row-standardized weights
W_mat <- listw2mat(W)                         # dense matrix if needed
# or spare:
W_sparse <- as(W_mat, "sparseMatrix")

saveRDS(nb,       "spatial_graph/nb.rds")
saveRDS(W,        "spatial_graph/W_listw.rds")
saveRDS(W_sparse, "spatial_graph/W_sparse.rds")
The nb object is your topological map — an adjacency list indexed by cell ID. It encodes the graph structure of ~9.6M polygons as a fixed reference. You build it once, serialize it, and never touch the geometry again.

Step 2 — Treat W as a linear operator on attribute vectors
For any variable x measured across all cells at time t, the spatially lagged value is just:

Wx_t  =  W_sparse  %*%  x_t
This is a sparse matrix-vector multiplication — O(n × avg_neighbors) instead of O(n²). For a queen contiguity graph on regular or near-regular cells, avg_neighbors ≈ 6–8, so this scales linearly with n.

For all variables at once (treating them as a matrix X of shape n × p):

r
# X_t : n_cells × n_variables matrix for year t
# W_sparse : n_cells × n_cells sparse adjacency

WX_t <- W_sparse %*% X_t    # produces spatial lags of ALL variables simultaneously
One matrix multiplication replaces n_cells × n_variables individual neighbor lookups.

Step 3 — Reformulate the full pipeline as tensor operations
Your data has structure: cells × years × variables — a 3D tensor. The spatial lag operation acts only on the cell dimension, so:

r
# Conceptually:
# Data tensor D[i, t, v] = value at cell i, year t, variable v
# Spatial lag: (WD)[i, t, v] = sum_j W[i,j] * D[j, t, v]

# In practice with data.table:
library(data.table)
library(Matrix)

dt <- as.data.table(joined_data)   # n_cells * n_years rows
vars <- names(dt)[!(names(dt) %in% c("id","year"))]

results <- lapply(unique(dt$year), function(yr) {
  X_t <- as.matrix(dt[year == yr, ..vars])          # n × p
  WX_t <- W_sparse %*% X_t                          # sparse matmul: n × p
  colnames(WX_t) <- paste0("slag_", vars)
  cbind(dt[year == yr, .(id, year)], as.data.table(WX_t))
})

spatial_lags <- rbindlist(results)
This replaces the cell-by-cell loop entirely.

Step 4 — Exploit the invariance for higher-order lags
If your downscaling uses second-order neighbors (neighbors of neighbors), these are also invariant:

r
W2 <- W_sparse %*% W_sparse    # compute ONCE
W3 <- W2 %*% W_sparse          # compute ONCE
Then apply to any year's data with the same matrix multiplications. The graph powers never change — only the attribute vectors do.

Step 5 — Precompute the neighborhood feature matrix
Instead of computing neighbor statistics (mean, sd, min, max of neighbors) on the fly during RF prediction, precompute them as fixed columns:

r
# For each year, augment the feature matrix with spatial summaries
augment_with_spatial_features <- function(X_t, W_sparse) {
  WX   <- W_sparse %*% X_t                   # neighbor means (row-standardized W)
  
  # Neighbor variance requires W with raw counts (not row-standardized):
  # Var = E[x²] - (E[x])²
  W_raw <- nb2listw(nb, style = "B")          # binary weights
  W_raw_sp <- as(listw2mat(W_raw), "sparseMatrix")
  n_neighbors <- rowSums(W_raw_sp)
  
  WX2  <- W_raw_sp %*% (X_t^2) / n_neighbors
  WX_var <- WX2 - WX^2
  
  cbind(X_t, WX, WX_var)
}
These augmented features go directly into predict(rf_model, newdata = ...) — no neighbor search at prediction time.

Why This Reduces Runtime from Weeks to Hours
Operation	Original	Reformulated
Neighbor search	9.6M × 14 years	Once
Spatial lag per variable	Cell loop × 110 vars × 14 yrs	14 sparse matmuls
Neighbor stats at prediction	Recomputed each call	Precomputed, stored
Memory pattern	Random access (neighbor lookup)	Sequential (matmul)
Parallelism	Hard (interdependent lookups)	Trivial (years are independent)
The sparse matrix multiplication is also BLAS-accelerated on any modern machine — it uses cache-friendly memory access patterns that the cell-by-cell loop cannot exploit.

The Conceptual Reframe
The key shift is recognizing that W is not a data transformation — it is a fixed linear map on your attribute space. Once you have it, applying it to new data is as cheap as a matrix multiply. The geometry problem is solved exactly once at the cost of building the graph; everything else is algebra.

In graph theory terms: you've computed the graph Laplacian (or its variants) once and stored it. All subsequent operations are just signal processing on a fixed graph — the same principle behind Graph Neural Networks, which solve exactly this class of problem at scale.
