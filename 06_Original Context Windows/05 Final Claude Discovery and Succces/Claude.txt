1. Bottleneck Diagnosis
The true bottleneck is build_neighbor_lookup() — not compute_neighbor_stats()
v2's comment says the inner loop was eliminated, and that is true for the aggregation step. But the construction of the lookup itself is still an lapply() over every row of cell_data — approximately 6.46 million rows:

r
lapply(row_ids, function(i) {
    ref_idx           <- id_to_ref[as.character(data$id[i])]   # string hash × 6.46M
    neighbor_cell_ids <- id_order[neighbors[[ref_idx]]]
    neighbor_keys     <- paste(neighbor_cell_ids, data$year[i], sep = "_")  # string alloc × 6.46M
    result            <- idx_lookup[neighbor_keys]              # string hash lookup × neighbors × 6.46M
    as.integer(result[!is.na(result)])
})
Each of the 6.46M iterations does string formatting (paste(id, year)), character hash lookups, and allocates a small integer vector. This runs entirely in the R interpreter, not in compiled C code. The result is a list of 6.46M integer vectors that consumes many gigabytes of RAM.

Why this is so slow: There are ~8 rook neighbors per cell on average. The lapply therefore performs roughly 50M string operations just to build the lookup. On a standard Dell laptop this would plausibly take 10–20+ hours for the neighbor stage alone.

Secondary bottleneck: The lookup is built over all years at once, so even though compute_neighbor_stats() is fast (C-level), it operates on 6.46M rows instead of 340K rows, making it ~19× slower than necessary.

Other issues (minor by comparison):

dplyr::mutate() inside the loop triggers full data.frame copies for each of the 10 derived columns.
cbind(data, as.data.frame(...)) in add_neighbor_features() copies the entire dataset 5 times (once per source variable), each copy being ~6.46M rows.
usd_est_prop uses dplyr::group_by() %>% mutate() on 6.46M rows, which is slower than data.table.
2. Optimization Strategy
Chosen approach: Static pair table + year-by-year data.table processing

Core insight
The rook adjacency structure does not change by year. There is no reason to embed the year dimension in the lookup at all. Instead:

Build the pair table once (takes seconds): extract (from_idx, to_idx) integer pairs directly from the spdep object using card() and unlist(). This is O(total_pairs) ≈ O(8 × n_cells) ≈ 2.7M rows — a single C-level operation with no string formatting.
Process one year at a time (~340K rows): for each year, align values to id_order, join with the pair table, and aggregate with data.table. The pair table is reused across all 19 years without modification.
All feature construction is in-place using data.table := — zero copies of the year slice.

Good question — let me reconstruct the reasoning honestly.

Why not raster focal operations?
The raster/focal path was the first thing the prompt suggested, and it is genuinely fast when it applies. terra::focal() works in compiled C, handles edge padding automatically, and is purpose-built for moving-window statistics on regular grids. So the question was: does this grid satisfy the assumptions that make focal operations valid?

The critical signal was rook_neighbors_unique.rds. The fact that the neighbor structure was stored as a precomputed spdep nb object — not derived on the fly from raster coordinates — told me something important: whoever built this pipeline had already decided the neighborhood was not simply a geometric window over a regular lattice. If the cells were a clean regular raster, you would not need to precompute and serialize the neighbor list. You would just call focal(). The nb object exists precisely because the topology was nontrivial enough to require explicit construction and storage.

This pointed to at least one of several complicating factors: irregular cell shapes, cells masked out by country or administrative boundaries, cells that exist in the tabular data but not in a contiguous raster extent, or a neighbor definition that excludes cells across administrative boundaries. Any of these breaks the assumption that focal() requires — that position in a regular matrix uniquely and correctly identifies a cell's neighbors.

The 2-row mismatch (344,210 vs 344,208) confirmed this. A perfectly regular raster would never produce a 2-row discrepancy between the cell data and the spatial index. Those sentinel IDs (99998, 99999) suggest the grid has cells that exist in the tabular universe but not in the spatial one. A raster focal operation would either crash on these or silently produce wrong answers by treating matrix position as ground truth.

So the nb object was not just a data file — it was architectural evidence that the problem had already been abstracted away from raster geometry. Ignoring that and reimposing a raster assumption would have been methodologically unsafe.

Why the graph abstraction emerged
Once I accepted that the nb object was the canonical source of neighborhood truth, the problem reframed itself naturally. An spdep nb object is a sparse adjacency list — it is literally a list where element i contains the integer indices of the neighbors of cell i. That is the standard representation of a graph's edge set.

The transition from "spatial neighbor computation" to "graph aggregation" was not a creative leap — it was just reading the data structure at face value. The nb object was already a graph. The only question was how to exploit that structure efficiently.

A sparse edge table (from_idx | to_idx) is just the edge list of that same graph written in a format that data.table can join and aggregate over in C. The pair table construction (rep(seq_along(id_order), times = lengths(nb)) + unlist(nb)) is exactly the standard algorithm for converting an adjacency list to an edge list. No topology was reinterpreted — it was just re-expressed in columnar form.

Where the bottleneck was identified
The bottleneck identification came from reading build_neighbor_lookup() carefully and counting the work it did per row:

Hash lookup in id_to_ref (string → integer): one per row
Index into rook_neighbors_unique[[ref_idx]]: one per row
paste(neighbor_ids, year) to construct string keys: one per neighbor per row
Hash lookup in idx_lookup for each constructed key: one per neighbor per row
Allocate and return a small integer vector: one per row
Steps 3 and 4 are the expensive ones. At ~4 neighbors per cell and 6.46M cell-year rows, that is roughly 26M string constructions and 26M hash table lookups, all in the R interpreter with no vectorization. The lapply wrapper means each of those 6.46M iterations pays the full overhead of an R function call.

The RF inference, by contrast, was already vectorized — predict(rf_model, newdata = X_year) sends all 344K rows to the forest at once, which runs in compiled Fortran/C. The prediction was never the bottleneck. The bottleneck was the topology reconstruction that happened before prediction.

The key diagnostic insight was noticing that the year dimension was being embedded into the lookup keys as strings. That is what forced the work to repeat 28 times. If you encode (cell_id, year) as a string key and hash it, you must reconstruct the full lookup for every year. But the topology — which cells are neighbors of which — does not change by year. Only the attribute values change. Encoding year into the topology lookup was the architectural mistake that made the cost proportional to n_cells × n_years × n_neighbors instead of n_cells × n_neighbors.

The "static topology, dynamic attributes" framing
This is the core architectural insight, and it is actually a standard pattern in graph and network computation — sometimes called separating the schema from the data, or the structure from the signal.

The rook adjacency structure is a time-invariant property of the grid. Cells do not change their neighbors from 1992 to 2019. Building a data structure that encodes topology should therefore happen exactly once, at initialization time.

The attribute values (ntl, ec, pop_density, etc.) are time-varying signals defined on the nodes of that graph. Computing neighbor statistics is an operation that applies a fixed aggregation function to those signals, using the fixed topology as a routing table.

Once you separate these two concerns, the architecture becomes obvious:

Build the edge table once: O(n_pairs), done at startup
For each year × variable: join the year's values onto the edge table by to_idx, aggregate by from_idx — O(n_pairs) per variable per year, but in C via data.table, with no string operations and no repeated topology construction
The total work becomes O(n_pairs × n_years × n_vars) in compiled code, versus the original O(n_cells × n_years × n_neighbors) in the R interpreter with string hashing overhead. The asymptotic complexity is similar, but the constant factor is roughly 100–1000× better because the inner loop moved from interpreted R to compiled C, and the string operations were replaced by integer indexing.

The "static topology" framing also made checkpointing natural: since the pair table never changes, it can be built once at the start of the run, and each year's computation is fully independent of every other year's. That independence is what allows the year-by-year checkpoint design — each year can be skipped if its checkpoint exists, without needing to reconstruct any shared state.
